Runtime Polymorphism Part 1

发表于 2019-03-17

Facebook folly.Poly如何做到双向绑定

​ 目前在研究C++的Runtime Polymorphism。这篇文章探讨folly.Poly库如何做到函数接口的双向绑定。

​

0. 起因

​ 研究Runtime Polymorphism的起因是看了Sean Parent关于Runtime Polymorphism的talk后，受益匪浅，在Sean Parent的talk中他实现了一种concept-model based polymorphism。

​ 出于好奇查阅了一下业界开源的实现，除了concept-model的模式，另外一种方式是函数指针的方式。

  • 在cpp conn的视频中，Louis Dionne介绍一种基于函数指针实现Runtime Polymorphism的方式，并将之开源在github dyno。

  • Facebook强大的开源库folly.Poly模块也提供了一种函数指针的方式。

1. folly.Poly库的实现

​ 基于对folly库的崇拜和偏爱，决定探究folly中实现。研究别人的库自然是从文档以及示例下手。folly文档给出的最简单的例子如下(只保留最基本的接口定义以及实现接口部分的代码)。

​ 在看源代码时，由于不清楚其中的设计逻辑，以及代码几乎全是模板，花了很长的时间也理不清设计和实现的必要性。

​ 比如我无法解释为什么Poly需要复杂到抓狂的Mine::Interface<PolyNode<Mine, PolyRoot<Mine>>>的设计，其中Mine是用户定义的接口。但是在代码的注释中写道，接口(示例代码中的Interface定义的函数)和实现(Circle&Square中实现的draw函数)是双向绑定的，但是Poly如何做到双向绑定困扰着我。

1.3 总结

根据代码分析以及编译错误，我们知道了Folly.Poly如何实现双向绑定。

  • Members中定义接口函数名。
  • 一种Object转化到接口时（例子中时Circle到drawble），使用MembersOf<I, T>，以此来绑定Members的函数和Object中的函数。
  • Interface实例化时使用MembersOf<I, I>来绑定Interface定义的函数和Members中的函数。
  • 通过Interface和Object分别向Members的函数绑定，实现了Interface和Object的绑定。

2. 局限性以及优势

其实虽然称作双向绑定，但不是一种一对一的绑定，只是去检查Members中定义的函数名，在Interface和Object中是否分别存在。
